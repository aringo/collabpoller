# collabpoller

`collabpoller` is a Python library and CLI for persistent polling, retention, and automation of interactions from privately operated Collaborator infrastructure.

It is intended for workflows where interactive testing is part of the initial phase, but interaction retrieval needs to continue over longer periods or be integrated into external scripting, automation, and analysis pipelines.

## Why this exists

Some testing workflows benefit from continued monitoring after the interactive phase has ended. `collabpoller` was built to support that operational need by making interaction polling scriptable, durable, and easier to integrate into long-running processes.

Common use cases include:

- long-duration polling of a private Collaborator deployment
- preserving and reviewing interaction history over time
- integrating interaction retrieval into Python-based automation
- supporting downstream analysis, alerting, and operational workflows

## Scope

This project is focused on persistent retrieval, logging, and automation. It is not aimed at replacing Burp’s interactive testing workflow.

## Design goals

- provide a scriptable interface for polling Collaborator interactions
- support long-term collection and retention of interaction data
- allow library-style use from Python applications
- make it easier to integrate Collaborator polling into external systems and operational tooling

## Typical workflow

A common workflow is:

1. use Burp during the active testing phase
2. generate or deploy payloads as part of that process
3. continue polling and retaining resulting interactions over time using `collabpoller`
4. consume the data in Python scripts, automation jobs, or other analysis pipelines

This allows interaction monitoring to continue after the interactive phase, without requiring an operator to manually keep a Burp session open solely for polling.

## Install

```bash
git clone https://github.com/aringo/collabpoller.git
cd collabpoller
pip3 install .
```

## First-time Setup

The default config path is:

```text
~/.config/collaborator_poller/config.json
```

### Fast path

If you already know your private Collaborator domain, initialize everything in one step:

```bash
collabpoller --init --subdomain mydomains.mycollab.com
```

That will:

1. create the default config if it does not exist yet
2. save the Collaborator domain
3. generate and store a unique client key for that operator

Expected output:

```text
[collabpoller] Created default config at /Users/<user>/.config/collaborator_poller/config.json
[collabpoller] Using subdomain provided on the command line.
[collabpoller] Client initialised.
  BIID: <your-biid>
  Key:  <your-base64-key>
```

### Walkthrough path

If you want the tool to generate a starter config first:

```bash
collabpoller --init
```

That creates a default config skeleton and tells you what to do next. Then either:

1. edit `~/.config/collaborator_poller/config.json` and set `"subdomain"`, then rerun `collabpoller --init`
2. or skip editing and run `collabpoller --init --subdomain mydomains.mycollab.com`

If two operators share the same private Collaborator server, each should run `--init` against their own config copy. Because keys are generated from `secrets.token_bytes(32)`, collisions are cryptographically negligible.

To rotate your key:

```bash
collabpoller --init --force
```

If your config does not already have a `subdomain`, add `--subdomain` there as well.

## Config Reference

Generated starter config:

```json
{
    "poll_interval": 45,
    "subdomain": "",
    "output": ""
}
```

Config with an initialized client:

```json
{
    "poll_interval": 45,
    "subdomain": "mydomains.mycollab.com",
    "output": "",
    "client": {
        "key": "<base64-32-bytes>",
        "counter": 1
    }
}
```

| Key | Description |
|---|---|
| `subdomain` | Your private Collaborator server domain |
| `poll_interval` | Seconds between polls when running in `--poll` mode |
| `output` | File to append raw JSON responses to; leave blank to skip logging |
| `client.key` | Base64-encoded 32-byte client key generated by `--init` |
| `client.counter` | Tracks payload generation so each new payload stays unique |

## Running CLI

### Generate a payload to use in a test

```bash
collabpoller --payload
```

Example output:

```text
Generated Payload: <subdomain>.mydomains.mycollab.com
```

### Poll once for interactions

```bash
collabpoller --check
```

### Poll continuously at the configured interval

```bash
collabpoller --poll
```

Example output:

```text
Checking for requests every: 45 seconds
2024-05-25 21:12:37: polling
```

### Use a non-default config file

```bash
collabpoller --config /path/to/config.json --poll
```

## Using in a script

```python
from collabpoller.poller import CollaboratorPoller

# Loads config; auto-inits key on first run if a subdomain is already set
poller = CollaboratorPoller()

# Generate a unique payload URL
payload = poller.gen_payload()
print(payload)

# Check for interactions once
poller.poll_all_biids()

# Start polling on the configured interval
poller.start_polling()
```
